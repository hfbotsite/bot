# Migrations (PostgreSQL)\n\nМиграции выполняются через Alembic.\n\nДиректория содержит:\n- `alembic.ini`\n- `alembic/` (env.py, versions)\n\n## Команды\n\nПрименить миграции:\n\n```bash\ncd services/migrations\nalembic -c alembic.ini upgrade head\n```\n\nСоздать миграцию (autogenerate):\n\n```bash\ncd services/migrations\nalembic -c alembic.ini revision --autogenerate -m \"your_message\"\n```\n\n## История ключевых миграций\n\n- `fe2642c1d02d_init.py` — стартовая схема платформы.\n- `cea880a6b903_margin_fields.py` — добавлены поля для явного учёта маржи/плеча:\n  - `positions.margin_mode`, `positions.leverage`, `positions.collateral_asset`\n  - `trade_fills.margin_mode`, `trade_fills.leverage`, `trade_fills.collateral_asset`\n- `044833282c53_lot_model.py` — добавлены таблицы для аудита лотов и распределения закрытий:\n  - `position_lots` — лоты (пакеты входа) по позиции\n  - `lot_allocations` — распределение закрывающих fills по лотам (включая `realized_pnl_gross`)\n\n## Правила хранения маржи/плеча (MVP)\n\n- `positions.*` — \"текущее/последнее известное\" состояние режима маржи и плеча по позиции.\n- `trade_fills.*` — снимок режима маржи и плеча на момент исполнения конкретного fill (если биржа возвращает эти данные).\n- Поля допускают `NULL`, так как разные биржи и режимы CCXT могут не возвращать leverage/margin_mode для каждого fill.\n\n## Правила лот-модели (MVP, Weighted Average)\n\n- Реализованный PnL в MVP считается по Weighted Average (средневзвешенная цена входа), как наиболее совместимый с динамическим усреднением.\n- `position_lots` ведётся для аудита/объяснимости и для возможного перехода на FIFO/LIFO в будущем.\n- `lot_allocations` фиксирует, какая часть закрывающего fill была отнесена к каким лотам. Это позволяет:\n  - восстановить историю частичных закрытий;\n  - объяснять рассчитанный realized PnL;\n  - сверять расчёты с отчётами биржи.\n
